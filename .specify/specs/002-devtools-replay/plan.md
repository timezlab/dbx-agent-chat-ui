# Implementation Plan: Dev-tools SSE Replay

**Branch**: `002-devtools-replay` | **Date**: 2026-07-02 | **Spec**: [spec.md](./spec.md)

**Input**: Feature specification from `.specify/specs/002-devtools-replay/spec.md`

## Summary

Add a developer/test-only **Replay** capability that drives the existing chat UI from a
recorded Responses-SSE stream **entirely in the browser** — no backend, no network — so the
same rendering path works in the static-export / single-file embed artifact where the dev mock
route does not exist. The sidebar **Dev tools** entry becomes a **Replay-mode toggle**; while on,
the chat composer is swapped for a **Replay control** (same footprint) offering source / play /
pause / delay / speed, and the area above the input is untouched.

Technical approach: extract the mock endpoint's frame-parsing and per-frame delay logic into a
**pure shared module** (`lib/stream/recording.ts`) reused by both the dev route and a new
client-side **`streamReplay`** stream source (`lib/stream/replay.ts`) that mirrors `streamSSE`'s
"exactly one terminal" contract while adding pause/resume/speed. `useChat` gains an isolated,
**non-persisted** replay conversation and drives the assistant turn through the *same*
`createResponsesParser` + `reduceStreamEvent` path as live streaming. The bundled default recording
ships as a **generated `.ts` module** (regenerated by build scripts) so the embed is self-contained;
a strip script regenerates the committed `default.txt` from a local capture with base64 chart
images removed. All Dev tools / Replay affordances are gated behind a non-secret
`NEXT_PUBLIC_DEV_TOOLS` flag, default **off**.

## Technical Context

**Language/Version**: TypeScript 5 / React 19 / Next.js 16.2.x (App Router, `output: "export"`).

**Primary Dependencies**: `@microsoft/fetch-event-source` (live only — replay uses no network),
`createResponsesParser` (shared parser), `reduceStreamEvent` (shared reducer), Tailwind v4 +
shadcn/Radix UI (Select, Button, Textarea, Tooltip), `@t3-oss/env-nextjs` + `zod` (env), Vite
(`vite-plugin-singlefile`) for the embed build.

**Storage**: None new. Replay state is ephemeral in-memory; replayed turns are **never** persisted
to `HistoryProvider` / localStorage (FR-020, SC-006).

**Testing**: `vitest` + Testing Library (unit + component), test-first per Principle VI. Node scripts
tested with small fixture strings (no reading the base64 capture).

**Target Platform**: Browser. Must build and run under `output: "export"` and the Vite single-file
embed.

**Project Type**: Single frontend project under `frontend/` (existing layout).

**Performance Goals**: Playback pacing matches the mock (text ~20 ms, tool ~400 ms) scaled by a speed
multiplier; pause must render **zero** frames; no busy-loop / hang on any timing input.

**Constraints**: UI-only, no backend/BFF/auth, static-export-safe, no secret in bundle, backends only
via the transport adapter boundary; every deployed file ≤ 9.5 MB (default recording stays tiny,
text-only).

**Scale/Scope**: ~1 shared module + 1 replay stream source + 2 build scripts + 1 generated module + 1
control component + `useChat`/context/config/env wiring + a mode toggle. Recordings are single-conversation
text files (low-MB upload cap).

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-check after Phase 1 design.*

| Principle | Verdict | Notes |
|-----------|---------|-------|
| I. UI-Only, No Backend | **PASS** | Replay runs fully client-side. The shared `recording.ts` module is **pure** (no `node:*`) so it is safe to import from client code; the mock route keeps its own `node:fs` I/O. No new server route. |
| II. No Secrets in Bundle | **PASS** | Only a new non-secret `NEXT_PUBLIC_DEV_TOOLS` selector flag; no credentials. |
| III. Static-Export Safe | **PASS** | Replay adds no route handler / server action / request-time dependency. Default recording is a generated `.ts` string inlined at build — no runtime fetch. Strip/gen scripts are build-time node scripts, never shipped. |
| IV. Backends Only Through Transport Adapters | **PASS (with note)** | Replay is **not a backend** and performs no fetch — it is a client-side *stream source* that reuses the single `createResponsesParser` + `reduceStreamEvent`, exactly like `mockTransport`/`streamSSE`. It is not forced through `ChatTransport.send()` because that contract returns a bare `AbortController` and cannot express pause/resume/speed; see Complexity Tracking. Components still never fetch or parse directly. |
| V. Customization Is a Contract | **PASS** | `ReplayControl` forwards `className` via trailing `cn(...)`, uses existing CSS-variable tokens and `data-slot` hooks; no hardcoded colors/radii. |
| VI. Test-First (NON-NEGOTIABLE) | **PASS** | Every non-trivial unit (shared module, `streamReplay`, `useChat` replay, config parse, control behavior, scripts) starts with a failing test. Enforced in `tasks.md`. |
| VII. Spec-Driven & Docs-as-Code | **PASS** | spec + plan + tasks under this folder; an ADR (`docs/design-docs/dev-replay.md`) records the replay-as-stream-source decision; mock-endpoint docs updated to reference the shared module. |

Deployment constraints: committed `default.txt` and the generated module stay well under 9.5 MB
(text-only, base64 stripped — SC-004). **No violations block Phase 0.**

## Complexity Tracking

| Violation / Bend | Why Needed | Simpler Alternative Rejected Because |
|------------------|------------|--------------------------------------|
| Replay is a stream source **outside** `ChatTransport.send()` (Principle IV nuance) | Pause/resume and live speed change are core to the feature (US2/US4); the `send() → AbortController` contract can only start/abort, so modelling replay as a `ChatTransport` would force the pause/resume state through a shape that cannot carry it. | Making replay a `ChatTransport` would either drop pause/resume (fails US2) or bloat the transport interface for every adapter with playback-only controls. Replay still reuses the **single** parser + reducer and never fetches/parses in a component, so the spirit of IV (one parse path, no component-level backend knowledge) holds. |
